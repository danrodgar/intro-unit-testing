\begin{frame}{Testing}

  \begin{itemize}
   \item \alert{Error}: A human mistake.
   \item \alert{Fault}: The result of a mistake, evidenced in some development or maintenance product.
   \item \alert{Failure}: A departure from the system's required behaviour.
  \end{itemize}

  \end{itemize}
  
  \end{frame}

\section{Assertions}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{What are Assertions?}

  An \alert{assertion} specifies a constraint over some state of computation. If it evaluates to false, it implies there exists an incorrect state in the program.
  
  Assertions: 
  
  \begin{itemize}
    \item can test whether the execution state is consistent at the point of examination.
    \item are predicates involving the state variables of a program in execution.
    \item are embedded in the program text.
  \end{itemize}
  
  Assertions are part of most programming languages.
  
  \end{frame}


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Assertions: History}


Hoare and Floyd started in the 60s (Hoare logic) the formal verification of programs. They introduced the concept of \emph{preconditions}, \emph{postconditions} and \emph{invariants}.

A triple describes how the execution of a piece of code changes the state of the computation:

  $[\{P\}C\{Q\}]$

\noindent where $P$ and $Q$ are assertions and $C$ is code.

\begin{itemize}
	\item $P$ precondition 
	\item $Q$ postcondition
\end{itemize}

In other words, a program behaviour can be verified using preconditions, postconditions and invariants.


\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{Assertions in Java}

  Assertions in Java are written as:

  \begin{lstlisting}[language=JAVA,basicstyle=\small]
  assert expression;
  \end{lstlisting}
  
  \texttt{assert} is a Java keyword and \texttt{expression} is any Boolean expression. When evaluated to \texttt{false}, an exception, \texttt{AssertionError}, will be raised. 
  
  They can also provide information for developers:
  
  \begin{lstlisting}[language=JAVA,basicstyle=\small]
  assert expression1 : expression2;
  \end{lstlisting}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{\texttt{Assert} Example}

\begin{lstlisting}[language=JAVA,basicstyle=\scriptsize]
  public double avgMarks(double[] marks){
 	// Pre-condition
	assert marks.length > 0 : "Array is empty!";
   	double avg = 0;
   	for (int i = 0 ; i < marks.length; i++){
	     //Invariant	
   	     assert marks[i] >= 0 && marks[i] <= 10 : "Mark is not valid!";
   	     avg += marks[i];
   	}
   	return avg/marks.length;
   }
\end{lstlisting}

\end{frame}



%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{\texttt{Assert} example}

\begin{lstlisting}[language=JAVA,basicstyle=\scriptsize]
public class AssertionExample {
  public double avgMarks(double[] marks){
   	assert marks.length > 0 : "Array is empty!";
   	double avg = 0;
   	for (int i = 0; i < marks.length; i++) {
   	     assert marks[i] >= 0 && marks[i] <= 10 : "Mark is not valid!";
   	     avg += marks[i];
   	}
   	return avg/marks.length;
   }

  public static void main(String args[]){
   	AssertionExample e = new AssertionExample();
   	double[] marks = {10, 5, 11, 3, 2, 7.5};
   	System.out.println(e.avgMarks(marks));
   	marks = new double[0];
   	System.out.println(e.avgMarks(marks));
   }
}
\end{lstlisting}


\end{frame}

\begin{frame}[fragile]{Assertions vs Exceptions}

Assertions are designed to check the program's correctness, i.e, as \emph{preconditions}, \emph{postconditions} or \emph{invariants}.

Assertions can be used to verify the state of a program but should NOT be used to:

\begin{itemize}
\item Validate arguments of public methods
\item Substitute the \textbf{exception handling} mechanims of a programming language
\item Modify values of attributes or objects
\item Be used in production code (assertions abruptly stop the execution of a program)
\end{itemize}

Exception handling is the process of responding to the occurrence of exceptions (usually in a graceful manner so that the execution is not interrupted). In Java, this is carried out with \texttt{try() ... catch() ... finally} statements.

Although most programming languages provide assertions, unit testing is usually carried out with frameworks such as JUnit (Java) or pytest (Python). 

\end{frame}

\subsection{JUnit 4 (Legacy)}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JUnit 4 (Legacy)}

JUnit 4 introduced annotations and static imports and is still relevant when migrating older test suites.
\begin{itemize}
  \item Static imports let us use assertion methods without qualifying the class name.
  \item JUnit 4 uses annotations instead of extending \texttt{TestCase}.
  \begin{itemize}
    \item Annotations are metadata consumed by the test runner.
    \item Naming conventions are no longer required, although they are still common.
  \end{itemize}
  \item Common legacy annotations include:
  \begin{itemize}
    \item Timeouts: \texttt{@Test(timeout=1000)}...
    \item Testing exceptions: \texttt{@Test(expected=DBConnectException.class)} 
    \item Ignore a test: \texttt{@Ignore("Ignore the following test for now")}
  \end{itemize}
  \item Legacy only: in JUnit Jupiter prefer \texttt{assertTimeout()}, \texttt{assertThrows()}, and \texttt{@Disabled}.
\end{itemize}

\end{frame}


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JUnit 4 -- Annotations}

\begin{itemize}
  \item \texttt{@Before} A method marked with \texttt{@Before} will be run before every test method. It is used to prepare the tests environment.
  \item \texttt{@After}
  \item \texttt{@Test} Identifies a test method.
  \item \texttt{@Ignore} The method under test will be ignored (to disable some tests which we do not want to repeat)
  \item \texttt{@AfterClass} Run only once after all test methods
  \item \texttt{@BeforeClass}
  \end{itemize}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\subsection{JUnit Jupiter (JUnit 5 and 6)}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JUnit Jupiter -- Annotations}

  \begin{itemize}
    \item \texttt{@BeforeEach} A method marked with \texttt{@BeforeEach} will be run before every test method. It is used to prepare the tests environment.
    \item \texttt{@AfterEach} It will be run after every test method.
    \item \texttt{@Test} Identifies a test method.
    \item \texttt{@Disabled} Disables a test or a test class.
    \item \texttt{@AfterAll} Run only once after all test methods
    \item \texttt{@BeforeAll} Run only once before all test methods
    \end{itemize}
  \end{frame}
 

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JUnit Jupiter -- Assert methods}

Assert methods are used to comparison between an expected value and an actual value to determine the success or failure of a test.
  
      \begin{itemize}
        \item \texttt{assertCONDITION(...)}
        \item \texttt{assertCONDITION(String message,...)} - The message is displayed when the assert method fails
      \end{itemize}
  
For example:

      \begin{itemize}
        \item \texttt{assertEquals(<> expected, <> actual)}
        \item \texttt{assertSame(...)} 
        \item \texttt{assertFalse(boolean condition)}
        \item \texttt{assertTrue(boolean condition)}
  \item \texttt{assertThrows(Exception.class, executable)}
  \item \texttt{assertAll(...)}
        %\item \texttt{assertNull(...)}
        %\item \texttt{assertNotNull(...)}
        \item Etc.
      \end{itemize}

Further information in the API: \texttt{org.junit.jupiter.api.Assertions}

%\url{https://junit.org/junit5/}
  
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JUnit Jupiter -- New functionality I}

  \begin{itemize}
    \item Custom display names - The \texttt{@DisplayName} annotation allows us to provide custom display names for test classes and methods, which can include spaces, special characters, and emojis.
  
    \item Nested test classes - Non-static nested classes (i.e., inner classes) can serve as \texttt{@Nested} test classes for logical grouping of test cases. 
  
    \item Tagging and filtering - It is possible to tag test classes and test methods with your identifiers using the \texttt{@Tag} annotation. 
  
    \item Meta-annotations and composed annotations - This allows us to create a custom composed annotation that will inherit the semantics of its meta-annotations.
  \end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JUnit Jupiter -- New functionality II}

  \begin{itemize}
    \item Dynamic tests - JUnit Jupiter introduces dynamic tests which are generated at runtime by a factory method annotated with the \texttt{@TestFactory} annotation.
  
    \item Repeated tests - Tests can be repeated a specified number of times by annotating the test method with the \texttt{@RepeatedTest} annotation.
  
    \item Parameterized tests - A test can be run multiple times with different arguments by annotating the test method with the \texttt{@ParameterizedTest} annotation.
    % and declaring at least one source to provide the arguments. 
  \end{itemize}

\end{frame}

\begin{frame}[fragile]{\texttt{ComplexNumber} Class -- Class Under Test (CUT)}

  We will use the \texttt{ComplexNumber} class as example, which will later be used to calculate quadratic equations.
  
  The structure of the \texttt{ComplexNumber} class is as follows:
  
  \begin{center}
   \includegraphics[width=75pt]{./figs/ComplexNumber}
  \end{center}
  
  The operations are as follows:
  
  \begin{itemize}
   \item Addition: $(a,b)+(c,d)=(a+c)+(b+d)i$
   \item Multiplication: $(a,b)\cdot(c,d)=(ac-bd) + (ad+cb)i$
   \item Equality: $(a,b)=(c,d) \Leftrightarrow a=c \wedge b=d$
  \end{itemize}
  
  \end{frame}


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%


\begin{frame}[fragile]{\texttt{ComplexNumber} class -- CUT I}


\begin{lstlisting}[language=Java,basicstyle=\scriptsize]
public class ComplexNumber {
  private double realPart;
  private double imagPart;

  public ComplexNumber(double re, double im) {
    realPart = re;
    imagPart = im;
  }

  public double getRealPart() {
    return realPart;
  }

  public double getImaginaryPart() {
    return imagPart;
  }

public ComplexNumber add(ComplexNumber c) {
  return new ComplexNumber(getRealPart() + c.getRealPart(),
                           getImaginaryPart() + c.getImaginaryPart());
}
\end{lstlisting}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{\texttt{ComplexNumber} class -- CUT II}

\begin{lstlisting}[language=Java,basicstyle=\scriptsize]
public ComplexNumber multiply(ComplexNumber c) {
  double re = getRealPart() * c.getRealPart()
           - getImaginaryPart() * c.getImaginaryPart();
  double im = getImaginaryPart() * c.getRealPart()
           + getRealPart() * c.getImaginaryPart();
  return new ComplexNumber(re, im);
}

@Override
public boolean equals(Object anObject) {
  if (anObject instanceof ComplexNumber) {
    ComplexNumber c = (ComplexNumber) anObject;
    return ((c.getRealPart() == getRealPart()) &&
            (c.getImaginaryPart() == getImaginaryPart()));
  } else
    return false;
  }
}
}
\end{lstlisting}

\end{frame}

\subsection{Testing Private Methods}

\begin{frame}{Testing Private Methods}

In general private methods are not tested directly (they are tested indirectly through other public methods). However, if there is a need to do there are several alternatives:

\begin{itemize}
 \item Change the visibility (make them public) temporarily. However, this changes the original program and does not follow the criteria of keeping tests and source code independently.
 \item Use Java reflection mechanisms
\end{itemize}

\begin{frame}[fragile]{Testing Private Methods - Example}

\begin{lstlisting}[language=Java,basicstyle=\tiny]
    public void testComputeDiscriminant() throws Exception {
        System.out.println("checkCoefficientsAreOk");
         Class c = null;
         Object ins = new QuadraticEquation(1,1,1);
         Method m = null;
         c = Class.forName("example.QuadraticEquation");
         m = c.getDeclaredMethod("computeDiscriminant");
         // Remove private access restriction:
         m.setAccessible(true);
         Object res = m.invoke(ins);
         Double result = (Double)res;
         System.out.println(result);
         double expectedResult = -3.0;
         assertEquals(result.doubleValue(), expectedResult);
    }
\end{lstlisting}

\begin{frame}{What are Mock objects?}


Sometimes tests need to be carried out without having all objects implemented or the instantiation of all needed classes is very expensive (e.g., when dealing with databases)

Advantages of using mock objects include:

\begin{itemize}
 \item Easy to instantiate
 \item If a CUT fails, the failure is located in the CUT, never in the mock!
 \item They allow us to reduce the testing time with simpler and faster classes
\end{itemize}

In Java, well-known frameworks include: Mockito, JMock and Easymock.

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Process}

Process to create mock objects

\begin{enumerate}
 \item Create the necessary Mock objects
 \item Define their behaviour, establishing expectations
 \item Create CUT instances using the mock objects
 \item Run the test methods 
 \item Check with the mock objects if expectations were fulfilled. 
\end{enumerate}

\end{frame}

\subsection{Behaviour-Driven Development}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{What is BDD?}

\alert{Behaviour-Driven Development (BDD)} is an agile development practice that encourages collaboration between developers, QA, and business stakeholders.

\begin{itemize}
  \item Derived from TDD --- shifts focus from \emph{how} to \emph{what} the system should do.
  \item Tests are written in plain natural language close to acceptance criteria.
  \item Bridges the gap between business requirements and executable tests.
  \item The most popular BDD framework for Java is \alert{Cucumber}.
\end{itemize}

\begin{block}{Key idea}
  Scenarios written in \emph{Gherkin} become both living documentation and automated tests.
\end{block}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{Gherkin Syntax}

Gherkin is a plain-English DSL for describing expected behaviour.

\begin{lstlisting}[language={}]
Feature: Search UAH degrees

    Given the user is located in the index page
    When  the user enters studies section
    And   the user clicks in the degree section

\begin{itemize}

  \item \texttt{Scenario} --- a single concrete test case.

\section{Code Coverage}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{What is Code Coverage?}

\alert{Code coverage} is a metric that measures what percentage of the source code is exercised by the test suite.

\begin{itemize}
  \item High coverage increases confidence that defects will be detected.
  \item Low coverage reveals areas of the code that are never tested.
  \item Coverage is \textbf{necessary but not sufficient} --- 100\% coverage does not guarantee correctness.
\end{itemize}

\begin{block}{Common coverage criteria}
  \begin{itemize}
    \item \alert{Line / Statement coverage} --- was each executable statement reached?
    \item \alert{Branch coverage} --- was each branch of every conditional taken?
    \item \alert{Method coverage} --- was each method called at least once?
  \end{itemize}
\end{block}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{JaCoCo}

\begin{block}{}
  \centering \url{https://www.jacoco.org/}
\end{block}

\alert{JaCoCo} (Java Code Coverage) is the standard code-coverage library for Java.

\begin{itemize}
  \item Instruments bytecode at runtime using a Java agent --- no source changes needed.
  \item Generates reports in HTML, XML, and CSV.
  \item Integrates with Maven, Gradle, and CI platforms (SonarQube, GitHub Actions).
  \item Reports line, branch, method, class, and cyclomatic-complexity metrics.
\end{itemize}

\end{frame}

\section{Mutation Testing}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{The Problem with Coverage}

Code coverage measures \emph{execution} --- not \emph{quality} of assertions.

\begin{exampleblock}{Counter-example: 100\% line coverage, zero assertions}
\begin{center}
\texttt{@Test void testAdd() \{ calc.add(1, 1); \}}
\end{center}
Every line of \texttt{add()} is executed, but the result is never checked.
\end{exampleblock}

\begin{itemize}
  \item A test that never asserts anything will achieve full coverage.
  \item \alert{Mutation testing} solves this by verifying your tests actually \emph{detect} faults.
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{What is Mutation Testing?}

\begin{block}{Idea}
  Automatically introduce a small \alert{mutation} (bug) into the source code, then re-run the test suite. If at least one test fails, the mutant is \alert{killed}. If all tests still pass, the mutant \alert{survived}.
\end{block}

\vspace{0.5em}
Common mutation operators:

\begin{itemize}
  \item Replace \texttt{+} with \texttt{-}, \texttt{*}, \texttt{/}.
  \item Replace \texttt{>} with \texttt{>=}, \texttt{==}.
  \item Negate conditions: \texttt{if (b == 0)} $\rightarrow$ \texttt{if (b != 0)}.
  \item Remove a method call or \texttt{return} statement.
\end{itemize}

\vspace{0.5em}
The \alert{mutation score} $= \dfrac{\text{killed mutants}}{\text{total mutants}} \times 100\%$

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Mutation Score vs.~Line Coverage}

\begin{center}
\begin{tabular}{lcc}
  \hline
  Scenario & Line coverage & Mutation score \\
  \hline
  No assertions     & 100\% & $\approx$0\%   \\
  Weak assertions   & 90\%  & 40--60\%        \\
  Strong assertions & 90\%  & $>$80\%         \\
  \hline
\end{tabular}
\end{center}

\vspace{0.5em}
\begin{itemize}
  \item High coverage + low mutation score $\Rightarrow$ tests are \emph{incomplete}.
  \item Aim for a mutation score $\geq 80\%$ for critical business logic.
  \item Some mutants are \emph{equivalent} (semantically identical to the original) and cannot be killed --- these are excluded from the score.
\end{itemize}

\end{frame}

\subsection{PIT (Pitest)}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{PIT -- Practical Mutation Testing for Java}

\begin{block}{}
  \centering \url{https://pitest.org/}
\end{block}

\begin{itemize}
  \item \alert{PIT} (Pitest) is the standard mutation testing tool for Java.
  \item Instruments bytecode --- no source changes needed.
  \item Integrates with Maven and Gradle.
  \item Much faster than earlier tools: reruns only tests \emph{relevant} to each mutant.
  \item Requires the \texttt{pitest-junit5-plugin} add-on for JUnit Jupiter/Platform.
  \item Generates an HTML report at \texttt{target/pit-reports/}.
\end{itemize}

\end{frame}

\begin{frame}[fragile]{PIT -- Example: Surviving Mutant}

The \texttt{Calculator} class has a \texttt{subtract} method:

\begin{lstlisting}[language=Java,basicstyle=\scriptsize]

    return a - b;
}
\end{lstlisting}

\texttt{CalculatorTest} has no test for \texttt{subtract}. PIT will replace \texttt{-} with \texttt{+}:

\begin{lstlisting}[language=Java,basicstyle=\scriptsize]
// mutant: return a + b;
\end{lstlisting}

All tests still pass $\Rightarrow$ \alert{mutant survived}. PIT flags this as a test gap.

Running PIT on the \texttt{PITExample} produces:
\texttt{Generated 11 mutations --- Killed 9 (82\%)}

The 2 surviving mutants indicate missing assertions ---
adding a \texttt{subtract} test will push the score to 100\%.

\end{frame}

\section{Testing Strategy and Quality}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Testing Levels and Test Pyramid}

Not all tests should run at the same level.

\begin{itemize}
  \item \alert{Unit tests}: fast, isolated, many.
  \item \alert{Integration tests}: verify collaboration between components.
  \item \alert{End-to-end tests}: validate user flows in a real environment.
\end{itemize}

\begin{block}{Test pyramid guideline}
  Prefer many unit tests, fewer integration tests, and a small set of E2E tests.
\end{block}

\begin{itemize}
  \item In this repository: \texttt{calculatorExample} and \texttt{complexNumber} are unit-focused.
  \item \texttt{CucumberTestExample}, \texttt{SeleniumTestExample}, and \texttt{PlaywrightExample} are UI/E2E-oriented.
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{What Makes a Good Test? (FIRST + AAA)}

\textbf{FIRST} properties:
\begin{itemize}
  \item \textbf{F}ast
  \item \textbf{I}ndependent
  \item \textbf{R}epeatable
  \item \textbf{S}elf-validating
  \item \textbf{T}imely
\end{itemize}

Recommended structure: \alert{AAA} (Arrange, Act, Assert)

\begin{lstlisting}[language=Java,basicstyle=\scriptsize]
@Test
void divideByZeroThrows() {
  // Arrange
  Calculator calc = new Calculator();
  // Act + Assert
  assertThrows(ArithmeticException.class,
      () -> calc.divide(1, 0));
}
\end{lstlisting}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Designing Better Test Cases}

Three techniques with strong payoff:

\begin{itemize}
  \item \alert{Equivalence partitioning}: divide input space into representative classes.
  \item \alert{Boundary value analysis}: test min/max and just inside/outside.
  \item \alert{Positive and negative paths}: valid scenarios and failure scenarios.
\end{itemize}

Example for \texttt{divide(a,b)}:
\begin{itemize}
  \item Typical case: \texttt{divide(10, 2)}
  \item Boundary/precision case: \texttt{divide(10, 3)}
  \item Error path: \texttt{divide(1, 0)}
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{Property-Based Testing}

Instead of only hand-picked examples, define \alert{properties} that should hold
for many generated inputs.

\begin{lstlisting}[language=Java,basicstyle=\scriptsize]
@Property
void addIsCommutative(@ForAll int a, @ForAll int b) {
    assertEquals(calc.add(a, b), calc.add(b, a));
}

@Property
void addZeroIsIdentity(@ForAll int a) {
    assertEquals(a, calc.add(a, 0));
}
\end{lstlisting}

\begin{itemize}
  \item On failure, the framework applies \alert{shrinking} to find the smallest
        counterexample.
  \item Java: \textbf{jqwik}. \quad Python: \textbf{Hypothesis}.
  \item Useful for algebraic laws, invariants, and round-trip properties.
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Avoiding Flaky Tests}

Flaky tests fail intermittently without code changes.

\begin{itemize}
  \item Avoid fixed sleeps; use explicit waits or Playwright auto-waiting.
  \item Keep test data deterministic and isolated.
  \item Do not depend on execution order.
  \item Control environment differences (browser, locale, timezone).
  \item For optional local browser runs, gate tests with:
  \texttt{-Drun.selenium=true}, \texttt{-Drun.playwright=true}.
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Test Doubles: Vocabulary}

\begin{itemize}
  \item \alert{Dummy}: passed around, never used.
  \item \alert{Stub}: provides predefined responses.
  \item \alert{Mock}: verifies interactions (calls, arguments, order).
  \item \alert{Spy}: wraps a real object and observes calls.
  \item \alert{Fake}: simplified working implementation (in-memory DB, etc.).
\end{itemize}

Use doubles to isolate the class under test and speed up feedback.

In this repository:
\begin{itemize}
  \item \texttt{MockitoATM} demonstrates Mockito-based mocks.
  \item \texttt{EasyMockExample} demonstrates EasyMock interactions.
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{CI Testing Strategy}

Suggested cadence in CI:

\begin{enumerate}
  \item Every commit: unit tests + JaCoCo.
  \item Pull requests: unit + selected UI smoke tests.
  \item Nightly or pre-release: full UI + PIT mutation testing.
\end{enumerate}

\begin{block}{Why this split?}
  Fast feedback for developers, deeper quality checks on a scheduled pipeline.
\end{block}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}[fragile]{CI Commands}

Use lightweight commands per stage:

\begin{lstlisting}[language={}]
# fast feedback (every commit)
cd code/calculatorExample
mvn test

# deeper feedback (scheduled)
cd code/PITExample
mvn test-compile org.pitest:pitest-maven:mutationCoverage
\end{lstlisting}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{Automated Testing: What to Automate First}

Automation should start with checks that are frequent, fast, and objective:

\begin{itemize}
  \item Run unit tests on every commit or before every push.
  \item Automate repeatable quality gates: compilation, linting, and coverage thresholds.
  \item Keep the fast test suite deterministic to avoid noisy failures.
  \item Reserve slower end-to-end or mutation suites for scheduled runs.
\end{itemize}

\begin{block}{Rule of thumb}
Automate \alert{execution} aggressively; automate \alert{judgment} carefully.
\end{block}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}{AI/Agents in Testing: Useful and Safe}

AI tools can accelerate testing work, but they do not replace engineering judgment.

\begin{itemize}
  \item Good use cases: generate test skeletons, propose edge cases, suggest property tests.
  \item Human review is mandatory: validate assertions, names, and intent.
  \item Watch for risks: plausible but wrong tests, over-mocking, brittle test logic.
\end{itemize}

Practical policy:
\begin{itemize}
  \item \alert{Automate execution}: yes.
  \item \alert{Automate authoring}: partially.
  \item \alert{Automate final quality decisions}: no.
\end{itemize}

\end{frame}

JUnit (or an equivalent framework) is standard practice in Java testing.

It can be used in combination with other frameworks such as Mockito, EasyMock, etc.

It should be part of the software development life cycle with:
\begin{itemize}
  \item Continuous integration
  \item Configuration management, e.g., Git.
  \item etc.
\end{itemize}

\end{frame}
